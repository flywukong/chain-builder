02 子系统 & AI 能力矩阵
子系统核心视图⚡ 默认 AI（云端）🔬 深挖（Codex）
📊 Monitor 大盘Gas/Latency/BlockRiver + 稳定性概览(slash/reorg) + 基础 MEV% 总览— 纯监控总览—
💎 MEV 分析Builder分布·MEV占比趋势·v1/v2(bidblock)·bid竞争·per-builder·local块占比T1 MEV格局/占比突变归因、某 builder 异常T2 结合 bid/miner 源码深挖
🌊 流量分析Gas趋势·TxPool·tx构成·gas price·大流量标注T1 异常大流量归因（MEV/合约事件/外部冲击）T2 读热点合约源码深挖成因
🔍 异常分析Slash·Reorg·慢块·漏槽时序（深度钻取）T1 slash/reorg/漏槽 根因 + 处置建议T2 读 parlia 共识源码定位根因
💾 存储节点磁盘用量·目录大小·增长趋势T1 增长异常分析 + 清理/扩容建议T2 结合 pruning/db 源码分析
🐛 Issues/Bounty每日拉 bnb-chain/bsc issues + ImmunefiT1 优先级评估 + 分类 + 摘要T2 读 bsc 源码 → triage/根因/修复思路
🔔 告警Keter urgent + 链上告警聚合T1 处理思路 + 值班指引T2 关联代码/历史告警深挖

T1（云端 API）：秒级流式，吃后端 gather 的结构化数据。T2（Codex CLI）：分钟级，代码感知，可读 bnb-chain/bsc 源码，用于需要"读代码才能答"的深挖。Issues 子系统的 T2 是 Codex 最大价值点 —— 它是编码 agent，能把 issue 对着源码分析。

03 AI 分析层设计（推理 provider + MCP 工具箱 + Codex）

不只吃预拉数据，AI 能在分析中按需查链上事实

所有子系统共用 AIAnalysisService：按子系统装载「配方（recipe）」拉一批种子上下文，再交给推理 provider 跑 tool-use 循环 —— provider 推理过程中可按需调用 MCP 工具查更多链上事实 / 文档，最后流式产出分析。这比"后端提前 gather 好全部数据"更强：AI 能顺着线索自己查。

接入官方 bnbchain-mcp（链上 read-only 查询）+ AskAi MCP（BNB docs/BEP/blog 语义检索）作为工具后端。MCP 仅服务 AI 层（v1.1/v1.2）—— 它的 get_latest_block 等是 RPC 轮询式，不替代 v1.0 的 WS newHeads 实时采集。

// 推理 provider（tool-use 循环）
interface Provider { run(prompt, seedCtx, tools) → AsyncIterable<token> }
CloudProvider  → 云端 AI API（Claude/OpenAI），默认，function-calling
CodexProvider  → "codex exec" headless, cwd=config.bscSource.path（深挖读码，原生 MCP）

// MCP 工具后端（read-only，挂给 provider 按需调）
tools = [
  BNBChainMcpTools,  // 官方 bnbchain-mcp: get_block/tx/receipt, read_contract/is_contract,
                     //   get_native_balance/get_erc20_*, get_chain_info
                     // ⚠ 只读：不配 PRIVATE_KEY + 工具白名单，禁用 transfer/write/approve
  AskAiMcpTools,     // BNB docs/BEP/blog 语义检索
]
// 后端做 MCP client：stdio 起子进程 或 SSE 连接，把工具暴露给 provider 的 tool-use

// 每子系统一份配方：种子上下文 + 可用工具 + 模板
recipes = {
  traffic:   { gather: gas趋势/tx构成/MEV, tools:[chain],       prompt: 大流量归因 },
  stability: { gather: slash/reorg/漏槽,    tools:[chain],       prompt: 根因 },
  storage:   { gather: 磁盘/目录/增长,      tools:[],            prompt: 存储分析 },
  issues:    { gather: issue正文+标签,       tools:[chain,askai], prompt: triage, deepRepo:"bnb-chain/bsc" },
  alerts:    { gather: keter告警,            tools:[chain,askai], prompt: 值班指引 },
}

// 路由
POST /api/ai/analyze   → CloudProvider.run(prompt, seedCtx, recipe.tools)  // T1 默认+工具
POST /api/ai/deepdive  → CodexProvider（cwd=源码）+ 同 ctx + 工具            // T2 深挖
// 结果按 (subsystem + context hash + 时间桶) 缓存
  • ▸provider 抽象：CloudProvider / CodexProvider 同一 run 接口（带 tools）；换云端模型或加 provider 不动业务
  • ▸MCP 工具箱：后端作 MCP client 接 bnbchain-mcp（stdio/SSE），AI 在分析中按需 get_block/get_transaction/read_contract/balance 查链上事实；AskAi MCP 查 BEP/docs/blog
  • ▸⚠ 只读安全（硬约束）：bnbchain-mcp 支持转账/合约写（需 PRIVATE_KEY）。AI 层绝不配 PRIVATE_KEY + 工具白名单只放只读，禁止 AI 触发 transfer/write/approve
  • ▸Codex 深挖：spawn codex headless，cwd=本地 bsc 源码（BSC_SOURCE_PATH，定期 git pull）；Codex 原生支持 MCP，可同时用上面工具箱
  • ▸流式 + 缓存：SSE 复用 <AIAnalysisPanel> 渲染；结果按 (subsystem+context hash+时间桶) 缓存；Codex 深挖加队列+进度态

05 与执行计划的关系
  • v1.0IA 重构（Home=Ring+Version → 7 子系统导航）+ Monitor/MEV/流量/异常/存储 的数据视图。采集层 WS 重写先行。AI 按钮先占位（灰）。
  • v1.1AIAnalysisService + CloudProvider（T1）+ bnbchain-mcp 只读工具上线，接 MEV/流量/异常/存储 的 ⚡ 一键分析；Fast Finality 监控。
  • v1.2CodexProvider（T2 深挖）+ AskAi MCP + Issues/Bounty 子系统（每日拉取 + Codex 读源码分析）；告警 AI 处理。

本架构图把 AI 的"位置"和"接法"定清楚，实现按 PLAN.md 分版本推进。provider + 工具抽象让 v1.1 先上云端+MCP、v1.2 再补 Codex/AskAi 不返工。MCP 只服务 AI 层，v1.0 实时采集仍走 WS newHeads。
